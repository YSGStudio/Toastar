## 1. 프로젝트 개요

### 1.1 추진 배경 및 목적

학생들이 자신의 그림·작품을 교내에서 안전하게 공유하고, 서로의 작품에 '좋아요'로 응원하며 창작 동기를 높이도록 돕는 **교내 전용 SNS**를 구축한다. 외부에 공개되지 않고 같은 학교 구성원만 이용하므로 학생 보호와 저작물 보호를 최우선으로 설계한다.

- 창작 결과물을 발표·공유할 수 있는 안전한 디지털 전시 공간 제공
- 좋아요(하트)와 주기별 시상으로 자발적인 참여와 성취감 유도
- 교사가 학급·기간·하트 정책을 직접 관리하는 운영 편의성 확보
- 작품 다운로드 차단 및 RLS 기반 접근 통제로 저작물·개인정보 보호

### 1.2 서비스 범위

| 구분 | 내용 |
|------|------|
| 이용 대상 | 우리 학교 교사 및 학생 (교내 전용, 외부 비공개) |
| 핵심 활동 | 작품 업로드 → 전시 → 좋아요 → 주기별 시상 |
| 지원 자료 | 이미지 파일, 외부 링크(URL), 동영상, 오디오(.wav) |
| 운영 단위 | 작품 업로드·기간·하트 정책은 학급 단위로 관리 |
| 열람 범위 | 가입·로그인한 모든 회원(전교 학생·교사)이 **전체 작품을 함께 열람** |
| 플랫폼 | 반응형 웹 (모바일·태블릿·PC 브라우저) |

### 1.3 핵심 특징 요약

- **간편 로그인**: 학생은 회원가입 없이 '학급코드 + 이름'만으로 로그인
- **주기 운영**: 교사가 설정한 기간 동안 학생 1인당 작품 1개 게시
- **한 줄 다중 전시**: 한 화면 한 줄에 여러 작품이 보이는 카드 그리드
- **전교 공개 열람**: 게시된 작품은 가입한 모든 회원(전교 학생·교사)이 함께 볼 수 있음
- **하트 정책**: 학생 1인 1일 하트 한도(기본 3개, 매일 자동 갱신) — 교사 조정 가능
- **주기별 시상**: 기간 종료 시 하트 최다 학생 발표 + 작품 카드 특수 효과 + 교사 기록 보관
- **안전 설계**: 다운로드 차단, 학생 작품 임의 수정·삭제 불가, RLS 보안 적용
- **로그인 시간 제한**: 교사가 평일·주말을 구분해 지정한 시간대에는 학생 로그인을 차단('로그인이 불가능합니다.' 안내)

## 3. 핵심 기능 명세

요구사항(1~12)을 기능 단위로 구체화한다. 각 항목은 화면·데이터·보안 설계(4~7장)와 연결된다.

### 3.1 계정 및 인증

**(1) 교사 회원가입 / 로그인**

- Supabase Auth의 이메일·비밀번호 방식으로 회원가입한다.
- 최초 로그인 후 학급을 생성하면 고유 '학급코드(class_code)'가 자동 발급된다.
- 교사 1명이 여러 학급을 운영할 수 있다(학급 단위로 데이터 분리).

**(2) 학생 계정 일괄 등록 (엑셀)**

- 교사가 학생 이름 목록이 담긴 엑셀 파일(.xlsx)을 업로드하면 학급 학생 계정이 한꺼번에 생성된다.
- 권장 엑셀 양식 예시 — 첫 행은 머리글, 한 행당 학생 1명:

  | 번호 | 이름 | 비고 |
  |------|------|------|
  | 1 | 김하늘 | |
  | 2 | 이서준 | |
  | 3 | 박지우 | (동명이인용) |

- 업로드 시 클라이언트에서 SheetJS(xlsx)로 파싱 → 서버(Route Handler)에서 `students` 테이블에 일괄 INSERT.
- **동명이인 처리**: 같은 학급 내 동일 이름이 있으면 가입 시 '번호' 또는 '구분 코드'를 함께 부여해 로그인 시 식별한다.

**(3) 학생 로그인 (학급코드 + 이름)**

- 학생은 회원가입 없이 학급코드와 이름만 입력해 로그인한다.
- 서버에서 (학급코드, 이름) 일치 여부를 확인한 뒤 `student_id`를 담은 세션 토큰을 발급한다(인증 모델은 7장 참고).

**(4) 학생 로그인 시간 제한 (교사 설정, 평일/주말 구분)**

- 교사가 환경설정에서 **평일(월~금)과 주말(토~일)을 구분해** 각각 로그인 차단 시간대(시작·종료 시각)와 사용 여부를 지정할 수 있다.
- 학생이 차단 시간대에 로그인을 시도하면 토큰을 발급하지 않고 **"로그인이 불가능합니다."** 메시지를 표시한다.
- 로그인 시점의 요일을 보고 평일/주말 규칙 중 해당하는 것을 적용한다(예: 평일 09:00~15:00 차단, 주말은 차단 없음).
- 차단 여부는 **서버(Route Handler)에서 토큰 발급 직전에** 현재 시각·요일과 설정 규칙을 비교해 처리한다(클라이언트 우회 불가).
- 자정을 넘어가는 시간대(예: 22:00~07:00)도 지원한다.
- **교사 로그인에는 적용되지 않는다**(교사는 항상 로그인 가능). 더 세분화된 요일별·특정 날짜 차단이 필요하면 같은 규칙 테이블을 확장한다.

```
// 서버(Route Handler) 학생 로그인 처리 의사코드
const dayType = isWeekend(오늘) ? 'weekend' : 'weekday';   // 토·일 = weekend
const rule = login_block_rules(class_id, dayType);          // 해당 구분 규칙 조회
if (rule && rule.enabled) {
  const now = 현재_시각;  // 학교 기준 시간대
  const inBlock =
    rule.start_time <= rule.end_time
      ? (now >= rule.start_time && now < rule.end_time)  // 같은 날 구간
      : (now >= rule.start_time || now < rule.end_time); // 자정 넘는 구간
  if (inBlock) return 거부("로그인이 불가능합니다.");
}
// 통과 시에만 학생 토큰 발급
```

### 3.2 작품 업로드

| 항목 | 내용 |
|------|------|
| 지원 형식 | 이미지(jpg/png/webp 등), 외부 링크(URL), 동영상(mp4 등), 오디오(.wav) |
| 게시 제한 | 활성 기간(주기)당 학생 1인 1개만 게시 (그림 1개 원칙) |
| 입력 정보 | 제목(필수, 짧게), 설명(선택, 간단히) |
| 썸네일 | 업로드 즉시 미리보기 썸네일 자동 생성 (이미지=리사이즈, 동영상=대표 프레임, 링크=대표 이미지/파비콘, 오디오=파형/기본 아이콘) |
| 저장 위치 | Supabase Storage 비공개 버킷에 저장, DB에는 경로·메타데이터 기록 |

> 주기당 1개 제한은 `(student_id, period_id)` 유니크 제약과 RLS 정책으로 이중 보장한다.

### 3.3 작품 열람 및 보호

- **전교 공개 열람**: 가입·로그인한 모든 회원이 전교 학생의 작품을 함께 본다(특정 학급에 한정되지 않음).
- **그리드 전시**: 한 줄에 여러 작품 카드가 배치되는 반응형 그리드(요구사항 9).
- **썸네일 표기 정보(요구사항 10)**: 작성자 이름, 작품 제목, 미리보기 이미지만 노출한다(설명은 상세보기에서).
- **확대 보기**: 썸네일을 누르면 모달이 열려 작품을 더 크게 보고 제목·설명·좋아요 수를 확인한다.
- **다운로드 차단(요구사항 2)**: 우클릭 저장·드래그 저장을 막고, 만료형(서명) URL과 워터마킹/오버레이로 원본 추출을 어렵게 한다.

### 3.4 좋아요(하트)

- 학생은 전교에 게시된 어떤 작품에든(및 정책에 따라 본인 작품) 하트를 줄 수 있다(요구사항 3).
- **일일 한도(요구사항 11)**: 학생 1인 1일 기본 3개. 자정(또는 학교 기준 시각)에 매일 자동 갱신된다.
- **교사 조정**: 환경설정에서 '일일 하트 한도'를 학급별로 변경할 수 있다.
- 같은 작품에 중복 하트는 불가(작품당 1회), 취소 허용 여부는 정책으로 선택한다.
- 한도 초과 시 버튼이 비활성화되고 '오늘 하트를 모두 사용했어요' 안내를 표시한다.

### 3.5 기간 운영 및 시상

**(1) 주기(기간) 설정**

- 교사가 작품 게시 주기의 시작일·종료일을 설정한다(요구사항 1·6).
- 주기는 연속적으로 운영되며, 새 주기를 시작하면 학생은 다시 1개의 작품을 올릴 수 있다.

**(2) 최신 자료 / 지난 자료 자동 분류 (요구사항 6)**

| 상태 | 기준 | 표시 위치 |
|------|------|-----------|
| 최신 자료 | 현재 활성 기간(시작일~종료일) 내 게시물 | '최신 자료' 탭 — 날짜 기준 최신순 갱신 |
| 지난 자료 | 종료된 기간의 게시물 | '지난 자료' 탭 — 기간 단위로 보관 |

> 기간 만료 시 게시물은 자동으로 '지난 자료'로 이동한다. 별도 이동 작업 없이, 게시물이 속한 `period`의 종료일과 현재 날짜를 비교해 분류하며, 필요 시 Supabase 예약 작업(pg_cron/Edge Function)으로 주기 종료·시상 집계를 자동 처리한다.

**(3) 시상 및 특수 효과 (요구사항 4)**

- 기간이 종료되면 해당 주기에서 하트를 가장 많이 받은 학생(들)을 발표한다.
- **특수 효과**: 최다 하트 작품 카드에 왕관 배지·반짝임(글로우)·테두리 애니메이션 등 시각 효과를 적용한다.
- **기록 보관**: 주기별 수상 결과(주기, 학생, 작품, 하트 수)를 `award_records` 테이블에 저장하고, 교사가 환경설정에서 누적 조회·관리한다.
- 동점자 처리(공동 수상) 및 표시 인원 수는 정책으로 설정한다(예: 상위 1~3명).
- 열람·하트가 전교 단위로 열려 있으므로, 시상 집계 범위(전교 통합 또는 학급별)는 교사가 선택할 수 있도록 한다.

### 3.6 작품 관리 (수정·삭제 권한, 요구사항 5)

- **교사**: 자기 학급 학생 작품을 수정·삭제할 수 있다(부적절 콘텐츠 관리 목적).
- **학생**: 한 번 올린 작품은 본인이 수정·삭제할 수 없다(신중한 게시 유도).

> 이 권한 차이는 UI에서 버튼 노출 제어뿐 아니라 RLS 정책(7장)으로 데이터 단에서 강제한다.

### 3.7 당겨서 새로고침 (요구사항 8)

- 모바일·터치 환경에서 목록 상단을 아래로 끌어내리면(Pull-to-Refresh) 최신 작품·하트 수를 다시 불러온다.
- 끌어내리는 동안 로딩 인디케이터를 표시하고, 임계치를 넘기면 데이터를 재요청한다.

## 4. 화면 설계 (UI/UX)

### 4.1 탭 구조 (요구사항 6)

| 탭 | 대상 | 내용 |
|------|------|------|
| 최신 자료 | 교사·학생 | 현재 활성 기간의 전교 작품 그리드 (최신 날짜순) |
| 지난 자료 | 교사·학생 | 종료된 기간의 전교 작품 보관함 (기간 선택 조회) |
| 환경설정 | 교사 전용 | 기간 설정, 일일 하트 한도, 학생 등록(엑셀), 시상 기록 관리 |

### 4.2 메인 그리드 (카드)

- 한 줄에 여러 작품이 들어가는 반응형 그리드: 모바일 2열 / 태블릿 3열 / PC 4~5열(요구사항 9).
- 각 카드(썸네일)에는 **미리보기 이미지 · 작품 제목 · 작성자 이름만** 노출(요구사항 10).
- 최다 하트 수상 카드에는 특수 효과(왕관·글로우)를 표시(요구사항 4).

**카드 예시 레이아웃**

```
├───────────────────────┤
│  작품 제목                  │

```

### 4.3 작품 상세 모달

- 썸네일 클릭 시 모달로 더 크게 표시(요구사항 2).
- 표시 정보: 큰 미리보기, 제목, 설명, 작성자, 좋아요 수, 좋아요 버튼.
- 동영상은 인라인 재생, 오디오(.wav)는 플레이어, 링크는 안전한 새 탭 열기.
- 다운로드·우클릭 저장 차단 적용(요구사항 2).

### 4.4 환경설정 화면 (교사 전용)

- **기간 관리**: 현재 주기 시작·종료일 설정, 새 주기 시작.
- **하트 정책**: 일일 하트 한도 입력(기본 3).
- **로그인 차단 시간**: 평일·주말을 구분해 각각 차단 시작·종료 시각과 사용 여부(토글)를 지정. 차단 시간에 학생이 로그인을 시도하면 "로그인이 불가능합니다." 안내.
- **학생 관리**: 엑셀 업로드로 일괄 등록, 학급코드 확인·재발급.
- **시상 기록**: 주기별 수상 결과 조회 및 보관.

### 4.5 시상 발표 화면

- 기간 종료 시 자동 노출: 최다 하트 학생·작품 강조 표시.
- 축하 애니메이션과 특수 효과 카드, 누적 수상 명단으로 연결.

## 5. 기술 아키텍처

### 5.1 기술 스택 (요구사항 7)

| 영역 | 기술 | 역할 |
|------|------|------|
| 프론트엔드 | React + Next.js (App Router) | 화면 렌더링, 라우팅, 서버 컴포넌트/Route Handler |
| 백엔드/DB | Supabase (PostgreSQL) | 데이터 저장, 인증, RLS 보안 정책 |
| 인증 | Supabase Auth + 커스텀 토큰 | 교사: 이메일 로그인 / 학생: 학급코드+이름 |
| 파일 저장 | Supabase Storage (비공개 버킷) | 이미지·동영상·오디오 저장, 서명 URL 제공 |
| 엑셀 처리 | SheetJS (xlsx) | 학생 명단 일괄 파싱·등록 |
| 스타일 | Tailwind CSS 등 | 반응형 그리드, 카드, 모달, 특수 효과 |
| 예약 작업 | Supabase pg_cron / Edge Function | 주기 종료·시상 집계·하트 카운터 갱신 |

### 5.2 데이터 흐름 개요

```
[브라우저 / React·Next.js]
        │  (1) 학생 로그인: 학급코드 + 이름
        ▼
[Next.js Route Handler]  ── 검증 ──▶ [Supabase: students 조회]
        │  (2) student_id 클레임이 담긴 세션 토큰 발급
        ▼
[Supabase PostgreSQL + RLS]  ◀── 모든 읽기/쓰기는 RLS 정책 통과
        │
        ▼
[Supabase Storage]  ── 서명 URL ──▶ 썸네일·원본(다운로드 차단)
```

> 교사는 Supabase Auth로 직접 인증되며(`auth.uid()`), 학생은 서버에서 검증 후 발급한 토큰의 클레임(`student_id`, `role`)으로 RLS의 적용을 받는다.

### 6.1 `teachers` (교사)

|------|------|------|
| id | uuid (PK) | auth.users.id 와 연결 |

### 6.2 `classes` (학급)

## 7. 보안 설계 (RLS, 요구사항 12)

### 7.1 인증 모델

- **교사**: Supabase Auth로 로그인 → `auth.uid()`가 `teachers.id`와 일치. `role='teacher'`.
- **학생**: 회원가입이 없으므로, 서버(Route Handler/Edge Function)가 (학급코드, 이름)을 검증한 뒤 `student_id`·`class_id`·`role='student'` 클레임이 담긴 Supabase용 토큰을 발급한다.
- RLS 정책은 `auth.uid()` 및 토큰 클레임(`auth.jwt()`)을 기준으로 행 접근을 통제한다.
- 교사·학생 모두 로그인 시 `authenticated` 역할을 가지며, **작품 열람(SELECT)은 모든 인증 회원에게 전교 단위로 허용**한다. 단, 업로드·수정·삭제·하트는 본인/교사 단위로 별도 제한한다.
- **로그인 시간 제한**: 학생 토큰 발급 직전, 서버가 요일(평일/주말)에 맞는 `login_block_rules` 규칙과 현재 시각을 비교한다. 차단 시간이면 토큰을 발급하지 않으므로 RLS 이전 단계에서 접근이 원천 차단된다(교사는 미적용).

> **핵심 원칙**: 모든 테이블에서 RLS를 활성화하고, 기본은 '차단'으로 두며 필요한 정책만 명시적으로 허용한다.

### 7.2 RLS 정책 예시

아래는 핵심 정책의 개념적 예시이다(실제 구현 시 함수·클레임 구조에 맞게 조정).

**(1) RLS 활성화**

```sql
alter table artworks       enable row level security;
alter table artwork_likes  enable row level security;
alter table students       enable row level security;
alter table periods        enable row level security;
-- 모든 테이블 동일하게 활성화
```

**(2) 작품 열람 — 가입 회원 전체 공개**

```sql
create policy "view_all_artworks_for_members"
on artworks for select using (
  -- 가입·로그인한 모든 회원(학생·교사)은 전교 작품을 열람 가능
  auth.role() = 'authenticated'
);
```

**(3) 작품 업로드 — 본인·활성 기간·주기당 1개**

```sql
create policy "student_insert_own_artwork"
on artworks for insert with check (
  student_id = (auth.jwt() ->> 'student_id')::uuid
  and (auth.jwt() ->> 'role') = 'student'
  and period_id in (select id from periods where status='active')
);
-- 주기당 1개는 UNIQUE(student_id, period_id) 제약으로 보장
```

**(4) 작품 수정·삭제 — 교사만 (요구사항 5)**

```sql
create policy "teacher_update_artwork"
on artworks for update using (
  class_id in (select id from classes where teacher_id = auth.uid())
);
create policy "teacher_delete_artwork"
on artworks for delete using (
  class_id in (select id from classes where teacher_id = auth.uid())
);
-- 학생에게는 update/delete 정책을 부여하지 않아 원천 차단
```

**(5) 좋아요 — 본인 + 일일 한도 (요구사항 11)**

```sql
create policy "student_insert_like"
on artwork_likes for insert with check (
  student_id = (auth.jwt() ->> 'student_id')::uuid
);
-- 일일 한도는 트리거로 검증:
--  INSERT 전 daily_heart_usage.used_count 와
--  classes.daily_heart_limit 를 비교하여 초과 시 거부,
--  통과 시 used_count 증가. (작품당 1회는 UNIQUE 제약)
```

### 7.3 파일 접근 및 다운로드 차단

- Storage는 **비공개(private) 버킷**을 사용하고, 표시는 짧은 만료의 서명 URL로만 제공한다.
- Storage 접근 정책으로 **가입·로그인한 회원이면 전교 작품 미리보기를 조회**할 수 있도록 허용한다(미인증 외부 접근은 차단).
- 프론트에서 우클릭·드래그 저장 차단, 직접 링크 노출 최소화, 필요 시 오버레이/워터마크로 원본 추출 난도를 높인다(완전 차단은 불가하므로 정책+기술 병행).

### 7.4 개인정보 보호 원칙

- 교내 전용 비공개 서비스로 검색엔진 색인·외부 공유를 차단한다.
- 학생 식별 정보는 이름 등 최소 정보만 수집하고 학급 범위 밖으로 노출하지 않는다.
